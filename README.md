   # Cyber Warfare Game - מתקיף נגד מגן

פרויקט גמר במגמת סייבר. משחק רשת בזמן אמת בין שני שחקנים - אחד תוקף, אחד מגן - עם שרת מרכזי שמשמש בתור חומת אש וגם בתור שופט.

---

## הסבר כללי על הפרויקט

זה משחק א-סימטרי (כלומר לשני הצדדים יש תפקיד שונה לגמרי):

- **המגן (Defender / Blue Team):** בונה חומת אש דרך GUI ב-pygame. מקליד חוקים (פורט לחסום, IP לחסום, מילה אסורה, גודל מקסימלי) ושולח אותם לשרת.
- **התוקף (Attacker / Red Team):** מריץ סקריפט Scapy שיורה חבילות UDP מזויפות לכיוון השרת, ומנסה לעקוף את החוקים.
- **השרת (Server):** מקבל את שני השחקנים ב-TCP, מסנפר את התעבורה ב-UDP על loopback, ובודק כל חבילה מול החוקים הפעילים. אם החבילה עברה - התוקף מקבל נקודה. אם נחסמה - המגן מקבל נקודה.

הראשון שמגיע ל-10 נקודות מנצח.

מתחת למכסה המנוע:
- TCP על פורט **8888** - לתקשורת ניהול (Handshake, חוקים, פידבק).
- UDP - לתעבורת התקיפה האמיתית. השרת בודק רק חבילות שמכילות payload שמתחיל ב-`GAME:` (חתימה של המשחק כדי שלא יסנן תעבורה אקראית של המחשב).
- DPI (Deep Packet Inspection) - השרת מסתכל על התוכן של החבילה ולא רק על הכותרת.

## תכנון פרוטוקול התקשורת

הקמתי פרוטוקול משלי בשכבת האפליקציה שרץ מעל TCP. הסיבה המרכזית: TCP זה stream, אז לפעמים שתי הודעות שנשלחות בנפרד מגיעות מודבקות (או חצי-חצי) בצד השני. בלי תוחם אי אפשר לדעת איפה הודעה אחת נגמרת והשניה מתחילה.

### Handshake (חיבור ראשוני)

מיד אחרי `connect()`, הקליינט שולח מחרוזת אחת בלבד שמזהה אותו:

- `ATK` - אני התוקף.
- `DEF` - אני המגן.

השרת מקבל בדיוק שני קליינטים - אחד מכל סוג. אם מישהו מנסה להתחבר שלישי, או ששולח משהו אחר, השרת מתעלם / סוגר את החיבור.

### תחביר חוקי החסימה (DEF -> Server)

המגן שולח חוקים בפורמט הבא, וכל חוק נגמר בתו `~`:

| פקודה | משמעות | דוגמה |
|--------|---------|--------|
| `P <port>~` | חסום פורט יעד UDP | `P 10000~` |
| `I <ip>~` | חסום כתובת IP מקור | `I 192.168.1.1~` |
| `W <word>~` | חסום חבילות שמכילות את המילה ב-payload | `W hack~` |
| `S*S <bytes>~` | קבע גודל מקסימלי לחבילה | `S*S 200~` |

### למה התו `~`?

זה ה-delimiter (התוחם) של הפרוטוקול. בלעדיו השרת לא יודע מתי הודעה נגמרת. בצד של השרת אני מצטבר את הבייטים בתוך buffer וקורא בלולאה:

```python
while "~" in buffer:
    rule, buffer = buffer.split("~", 1)
    parse_rule(rule)
```

ככה גם אם שתי הודעות הגיעו ביחד או שאחת הגיעה חתוכה לשניים - אין בעיה, מחכים עד שיש `~` ואז מעבדים.

בחרתי דווקא ב-`~` כי הוא תו שלא מופיע באף שם של פורט / IP / מילה רגילה, אז אין סיכוי שיתבלבל עם תוכן.

### הודעות פידבק מהשרת

אחרי שכל חבילת UDP נבדקת מול ה-rule engine, השרת שולח חזרה לשני השחקנים ב-TCP:

- `SCORED~` - החבילה נחסמה -> נקודה למגן.
- `MISSED~` - החבילה עברה -> נקודה לתוקף.

(גם הפידבק נגמר ב-`~` כי משתמשים באותו פרוטוקול לשני הכיוונים.)

### תנאי ניצחון (Broadcast)

ברגע שאחד הצדדים מגיע ל-10 נקודות, השרת שולח הודעת סיום לשניהם:

- `DEFENDER WINS~` - המגן ניצח (חסם 10 חבילות).
- `ATTACKER WINS~` - התוקף ניצח (10 חבילות עברו את חומת האש).

אחרי השידור הזה השרת סוגר את החיבורים והמשחק מסתיים.

## מדריך למשתמש


### סדר הפעלה (חשוב!)

צריך להריץ בשלושה טרמינלים נפרדים, בסדר הבא:

1. **שרת** - הוא צריך להיות למעלה ראשון, אחרת אין למי להתחבר:
   ```
   python Server.py
   ```
2. **מגן** - פותח את ה-GUI:
   ```
   python Defender_GUI.py
   ```
3. **תוקף** - מתחיל לירות חבילות:
   ```
   python Attacket.py
   ```

אפשר להריץ הכל באותו מחשב (loopback / 127.0.0.1) או על מחשבים שונים ברשת - פשוט לשנות את ה-IP של השרת בקבועים של הקליינטים.

### איך משחקים

**אם אתה המגן:**
- חלון ה-pygame ייפתח עם שלושה שדות קלט: Target Port, Banned IP, Forbidden Word.
- לוחצים על השדה כדי לשים עליו פוקוס, מקלידים את הערך, ואז לוחצים על כפתור ה-"BLOCK!" המתאים.
- החוק נשלח לשרת ומתחיל לחסום מיידית.
- ה-LIVE RADAR בצד מראה בזמן אמת אם חסמת חבילה (SCORED) או שמשהו עבר לך (MISSED).
- המטרה: להגיע ל-10 חסימות לפני שהתוקף מעביר 10 חבילות.

**אם אתה התוקף:**
- הסקריפט רץ אוטומטית ויורה חבילות UDP עם payload שמתחיל ב-`GAME:`.
- כל חבילה שעוברת את חומת האש - נקודה לך.
- המטרה: לעקוף את החוקים של המגן (לשנות פורט, לשנות מילה, להקטין חבילות) עד שתגיע ל-10.
